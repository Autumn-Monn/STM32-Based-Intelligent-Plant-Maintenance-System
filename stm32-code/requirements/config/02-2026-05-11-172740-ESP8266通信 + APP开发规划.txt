  需要讨论的关键决策

  1. USART 资源冲突（最关键）

  USART1 同时被 debug_uart 和 ESP8266 占用，而其他 USART 引脚全部冲突：
  - USART2（PA2 TX / PA3 RX）→ PA2 = 风扇继电器，PA3 = 蜂鸣器
  - USART3（PB10 TX / PB11 RX）→ PB10 = ESP8266 RST

  可选方案：

  ┌──────────────────────────┬─────────────────────────┬───────────────────┐
  │           方案           │          描述           │       优劣        │
  ├──────────────────────────┼─────────────────────────┼───────────────────┤
  │ A. USART1 给             │ 最简单，调试改用 OLED   │ 丢失串口调试能力  │
  │ ESP8266，砍掉 debug_uart │ 或 SWO                  │                   │
  ├──────────────────────────┼─────────────────────────┼───────────────────┤
  │ B. USART1 给             │ 保留调试输出（仅 TX，无 │ 需要 ST-Link SWO  │
  │ ESP8266，debug 改 SWO    │  RX）                   │ 支持              │
  │ (PB3)                    │                         │                   │
  ├──────────────────────────┼─────────────────────────┼───────────────────┤
  │ C. USART3 部分重映射     │ F103C8T6 没有           │ 不可行            │
  │ (PB10→PC10/PC11)         │ PC10/PC11（仅48脚）     │                   │
  ├──────────────────────────┼─────────────────────────┼───────────────────┤
  │ D. USART1 分时复用       │ 调试和 ESP8266          │ 复杂，容易出错    │
  │                          │ 共用，状态机切换        │                   │
  └──────────────────────────┴─────────────────────────┴───────────────────┘

  我倾向方案 A：ESP8266 接入后，debug_uart
  模块降级为空操作或直接移除。系统已经有 OLED 显示状态，调试阶段可以先不接
  ESP8266 而继续用串口。你怎么看？

  2. AT 指令引擎——非阻塞改造

  Demo 的阻塞式 Delay_ms() + ESP_WaitResp() 不能直接搬过来。需要改造为状态机：

  ESP_STATE_IDLE → ESP_STATE_SENDING → ESP_STATE_WAIT_RESP → ESP_STATE_OK /
  ESP_STATE_TIMEOUT

  - USART1 RX 中断收数据，存入环形缓冲区（类似现有 debug_uart 但更大，建议
  256~512 字节）
  - esp8266_task() 在主循环中调用，用 HAL_GetTick() 做超时检测
  - 初始化序列（WiFi 连接 + MQTT 连接）用步骤编号驱动，每步完成后进入下一步

  3. OneNET 云平台侧

  Demo 使用的是 OneNET 物模型（新版） MQTT 接入：
  - 服务器：mqtts.heclouds.com:1883
  - 认证：AT+MQTTUSERCFG 中的 token（产品ID + 设备名 + 签名）
  - 上报 topic：$sys/{product_id}/{device_name}/thing/property/post
  - 下发 topic：$sys/{product_id}/{device_name}/thing/property/set
  - 数据格式：OneNET 物模型 JSON

  你需要在 OneNET 控制台完成：
  1. 创建产品 → 获取 product_id
  2. 创建设备 → 获取 device_name + 计算 token
  3. 定义物模型属性：temperature (float)、humidity (float)、pump_state
  (bool)、fan_state (bool)、mode (enum)

● VS Code 本身没有直接打包 APK 的扩展。但有几条路线可以实现全程 VS Code：

  方案 1：Vue/React + Capacitor（推荐）

  Capacitor 是 Ionic 团队出的工具，把 Web 应用包一层原生壳变成
  APK，全程命令行操作：

  # 1. 创建 Vue 项目并开发完成后
  npm run build

  # 2. 添加 Capacitor
  npm install @capacitor/core @capacitor/cli
  npx cap init

  # 3. 添加 Android 平台
  npm install @capacitor/android
  npx cap add android

  # 4. 同步 web 产物到 Android 工程
  npx cap sync

  # 5. 打包 APK（需要安装 Android Studio，但只用它的命令行工具）
  cd android && ./gradlew assembleDebug

  产出：android/app/build/outputs/apk/debug/app-debug.apk

  优势：VS Code 写代码，命令行打包，不依赖 HBuilderX。Android Studio
  装一次就行（主要是为了 Android SDK）。

  方案 2：UniApp CLI 开发 + HBuilderX 只用一次打包

  VS Code 中写所有代码 → 最终 HBuilderX 打开项目点一次"云打包" → 拿到
  APK。HBuilderX 只是个打包工具，不需要在里面写代码。

  方案 3：直接做 PWA（最省事）

  Progressive Web App——手机浏览器打开网址 → "添加到主屏幕" → 图标和体验跟 APP
  几乎一样，零打包。缺点是答辩时说"这是个网页"可能不如"这是个 APP"有说服力。
