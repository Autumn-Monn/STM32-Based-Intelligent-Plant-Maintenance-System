# 阶段9：联动控制与系统集成 — 实施计划

## 1. 阶段目标

将已验证的各独立模块（传感器采集、执行器控制、OLED 显示、按键输入、数据存储）整合为一个闭环控制系统，对齐设计要求文档 `01-智能盆栽养护系统设计要求.md` 中的功能要求 1.(1)~(3) 及 5.(2)。

## 2. 对齐设计要求

### 2.1 功能要求 1.(1) 环境监测

| 要求 | 现状 | 阶段9 动作 |
|------|------|-----------|
| 实时采集土壤湿度 | `soil.c` ADC 采集已打通 | 在 `control.c` 中统一周期采样，缓存最新值 |
| 实时采集土壤温度 | `ds18b20.c` 非阻塞采集已打通 | 直接调用 `ds18b20_get_cached_raw()` |
| 湿度方向 | 干燥时 ADC 值小，湿润时 ADC 值大 | 阈值比较：当前值 < 下限 → 缺水 |

### 2.2 功能要求 1.(2) 自动化控制

新建 `control.c / control.h`，实现带滞回的闭环控制：

| 条件 | 动作 | 恢复条件 |
|------|------|---------|
| 湿度 < 下限阈值 | 启动水泵 (PA1) | 湿度 ≥ 上限阈值时关闭 |
| 温度 > 上限阈值 | 启动风扇 (PA2) | 温度 ≤ (上限 - 滞回量) 时关闭 |
| 湿度 < 报警阈值（极干） | 蜂鸣器报警 + 红灯 | 湿度回升至下限以上 |
| 温度 > 报警阈值（极热） | 蜂鸣器报警 | 温度回落至上限以下 |

手动模式：按键直接控制水泵/风扇，自动逻辑暂停。

### 2.3 功能要求 1.(3) 用户交互

#### 三色 LED 指示

| 状态 | LED |
|------|-----|
| 正常 | 绿色常亮 |
| 缺水（湿度低于下限） | 红色常亮 |
| 低温（预留，温度低于某值） | 蓝色常亮 |
| 报警（极端干燥或超温） | 红色闪烁 |

#### OLED 显示（运行模式）

```
Row 0: 温度: 26.5C        (96px, 余32px)
Row 1: 湿度: 2130          (80px, 余48px)
Row 2: 自动:P:OFF F:OFF    (128px, 刚好满)
Row 3: 状态:正常            (72px, 余56px)
```

> `°` 符号暂不显示，直接用 `C` 表示摄氏度。

#### OLED 显示（设置模式）

```
Row 0: == 设置 ==           (80px)
Row 1: > 植物: 常规         (96px)
Row 2:   阈值: 1800         (96px)
Row 3: K1:下项 K4:保存      (120px, 余8px)
```

#### 蜂鸣器报警

极端干燥或超温时触发非阻塞间歇鸣叫（如 300ms 响 / 300ms 停）。按 K4 可消除报警。

#### 植物类型自适应阈值

支持 3 组预设，用户可在设置模式中选择后逐项微调：

| 类型 | 湿度下限 | 湿度上限 | 温度上限(°C) | 温度报警(°C) | 湿度报警 |
|------|---------|---------|-------------|-------------|---------|
| 耐旱 | 1400 | 1800 | 38 | 43 | 1000 |
| 常规 | 1800 | 2200 | 35 | 40 | 1200 |
| 喜湿 | 2200 | 2800 | 32 | 37 | 1600 |

### 2.4 功能要求 5.(2) 历史数据存储

- 已有 `data_store.c` 每 5 分钟将温度 + 湿度写入 AT24C02 环形缓冲区（最多 63 条）
- 本阶段保持不变，继续在主循环中调用 `data_store_task()`

## 4. 按键逻辑

### 4.1 系统模式

| 模式 | 说明 |
|------|------|
| 运行-自动 | 根据阈值自动控制执行器 |
| 运行-手动 | 按键直接控制水泵/风扇，自动逻辑暂停 |
| 设置 | 调整植物类型、各项阈值 |

### 4.2 运行模式按键

| 按键 | 短按 | 长按（≥2秒） |
|------|------|-------------|
| K1 | 自动 ↔ 手动 切换 | 进入设置模式 |
| K2 | 手动模式下切换水泵（自动模式下无效） | — |
| K3 | 手动模式下切换风扇（自动模式下无效） | — |

### 4.3 设置模式按键

| 按键 | 功能 |
|------|------|
| K1 | 切换到下一个设置项 |
| K2 | 当前项减小 |
| K3 | 当前项增大 |
| K4 | 保存至 AT24C02 并退出设置模式 |

### 4.4 设置项列表

| 序号 | 设置项 | 默认值（常规型） | 步进 | 范围 |
|------|--------|----------------|------|------|
| 0 | 植物类型 | 常规 | 整组切换 | 耐旱 / 常规 / 喜湿 |
| 1 | 湿度下限（启动水泵） | 1800 | ±100 | 500 ~ 3500 |
| 2 | 湿度上限（停止水泵） | 2200 | ±100 | 800 ~ 3800 |
| 3 | 温度上限（启动风扇, °C） | 35 | ±1 | 20 ~ 50 |
| 4 | 温度报警阈值 (°C) | 40 | ±1 | 25 ~ 55 |
| 5 | 湿度报警阈值（极干） | 1200 | ±100 | 300 ~ 2500 |

> 选择植物类型时，一次性加载该组全部阈值；之后用户仍可逐项微调。

### 4.5 长按检测

- 当前 `key.c` 仅支持短按事件，需要新增长按事件
- 长按判定时间：**2 秒**
- 新增事件枚举：`KEY_EVENT_1_LONG`（仅 K1 需要长按）

## 5. OLED 字模方案

### 5.1 现有字模

| 枚举 | 汉字 |
|------|------|
| HZ_WEN | 温 |
| HZ_DU | 度 |
| HZ_SHI | 湿 |

### 5.2 需要新增的汉字（24 个）

| 用途 | 汉字 |
|------|------|
| 模式显示 | 自、动、手 |
| 状态显示 | 状、态、正、常 |
| 报警状态 | 缺、水、高 |
| 设置界面 | 设、置 |
| 植物类型 | 植、物 |
| 类型名称 | 规、喜、耐、旱 |
| 阈值设置 | 阈、值 |
| 导航提示 | 下、项、保、存 |

> 总计 27 个汉字（含已有 3 个），每个 32 字节，共 864 字节 Flash 占用。

### 5.3 字模生成方式

由用户使用 **PCtoLCD2002** 取模：

- 点阵大小：16×16
- 取模方式：**列行式**
- 取模走向：**逆向**（低位在前）
- 阴码/阳码：**阴码**
- 字体：**宋体**
- 输出格式：C51 格式（十六进制数组）

操作流程：
1. 在 PCtoLCD2002 中依次输入 24 个新增汉字
2. 生成字模数据
3. 复制粘贴到 `User/Inc/oled_font.h` 的 `OLED_HZ16X16` 数组中
4. 同步更新 `oled.h` 中的 `hz_index_t` 枚举

## 6. 阈值掉电保存方案

### 6.1 AT24C02 存储布局

当前 `data_store.c` 使用的地址空间：

| 地址 | 内容 |
|------|------|
| 0x00 ~ 0x03 | data_store header (4 bytes) |
| 0x04 ~ 0xFF | data_store records (63 × 4 = 252 bytes) |

AT24C02 总共 256 字节（0x00 ~ 0xFF），已被 data_store 完全占用。

**方案：从 data_store 尾部划出 16 字节用于阈值存储。**

将 `STORE_MAX_RECORDS` 从 63 减少到 59（释放 4×4=16 字节），在高地址段存放阈值配置：

| 地址 | 内容 | 大小 |
|------|------|------|
| 0x00 ~ 0x03 | data_store header | 4 bytes |
| 0x04 ~ 0xF3 | data_store records (59 × 4) | 236 bytes |
| 0xF4 | 阈值配置 magic (0xCF) | 1 byte |
| 0xF5 | 植物类型 (0/1/2) | 1 byte |
| 0xF6 ~ 0xF7 | 湿度下限 (uint16_t LE) | 2 bytes |
| 0xF8 ~ 0xF9 | 湿度上限 (uint16_t LE) | 2 bytes |
| 0xFA | 温度上限 (uint8_t, °C) | 1 byte |
| 0xFB | 温度报警 (uint8_t, °C) | 1 byte |
| 0xFC ~ 0xFD | 湿度报警 (uint16_t LE) | 2 bytes |
| 0xFE ~ 0xFF | 保留 | 2 bytes |

> 共 12 字节有效数据，上电时读取，magic 不匹配则加载默认值（常规型）。

## 8. 主循环结构

```c
/* USER CODE BEGIN 2 */
led_init();
key_init();
relay_init();
debug_uart_init();
soil_init();
ds18b20_init();
oled_init();
data_store_init();
control_init();        // 从 AT24C02 加载阈值，初始化为自动模式
/* USER CODE END 2 */

while (1)
{
    key_scan();
    control_key_handler();   // 按键事件分发 → 模式切换/手动控制/设置调整
    ds18b20_task();          // 非阻塞温度采集
    control_task();          // 采集→阈值判断→执行器→LED→蜂鸣器
    oled_display_task();     // 根据当前模式显示对应界面
    data_store_task();       // 历史数据周期存储
}
```

> 移除现有 main.c 中的串口命令交互（`while (debug_uart_has_data())` 段）和 `i2c_test_task()` 调用。串口日志保留为 `control.c` 内部的状态变化打印。

## 9. 实施步骤

按以下顺序逐步实施，每步可独立编译验证：

### Step 1：key.c 长按支持

- 新增 `KEY_EVENT_1_LONG` 枚举
- 修改 `key_scan()` 内部逻辑，K1 增加按下计时
- 验证：K1 短按串口打印 short，长按 2 秒串口打印 long

### Step 2：data_store 阈值存取

- 减少 `STORE_MAX_RECORDS` 到 59
- 新增 `threshold_config_t` 结构体
- 新增 `data_store_load_thresholds()` / `data_store_save_thresholds()`
- 验证：通过串口命令触发读写，断电后重启检查恢复

### Step 3：control.c 核心逻辑

- 定义系统模式枚举（AUTO / MANUAL / SETTINGS）
- 实现 `control_init()`：加载阈值，初始化为 AUTO 模式
- 实现 `control_task()`：采集 → 阈值比较 → 执行器控制 → LED 指示 → 蜂鸣器
- 实现 `control_key_handler()`：运行模式和设置模式下的按键分发
- 验证：自动模式下模拟干燥/高温，观察继电器和 LED 响应

### Step 4：OLED 字模与显示升级

- 用户使用 PCtoLCD2002 生成 24 个汉字字模，粘贴到 `oled_font.h`
- 扩展 `hz_index_t` 枚举
- 改写 `oled_display_task()`：运行模式4行 + 设置模式4行
- 验证：运行模式与设置模式切换，显示内容正确

### Step 5：main.c 整合与联调

- 主循环改写为最终结构
- 移除旧的串口命令交互和测试代码
- CMakeLists.txt 加入 `control.c`
- 全流程验证：上电 → 自动控制 → 按键切手动 → 长按进设置 → 调阈值 → 保存退出 → 掉电重启恢复
