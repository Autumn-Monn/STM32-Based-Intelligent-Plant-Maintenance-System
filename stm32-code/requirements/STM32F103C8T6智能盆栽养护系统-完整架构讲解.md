## 第一节：全局概览

### 1.1 系统目标

这是一套基于 STM32F103C8T6 的**智能盆栽养护系统**，核心功能闭环为：

```
传感器采集 → 控制决策 → 执行器动作 → 状态显示 + 云端上报 + 远程控制
```

具体来说：

- **自动监测**：通过 FC-28 土壤湿度传感器 (ADC) 和 DS18B20 温度传感器 (One-Wire) 实时采集环境数据
- **自动控制**：根据可配置的阈值，带滞回地控制水泵（灌溉）和风扇（散热），两个回路独立运行
- **人机交互**：0.96 寸 OLED 显示温湿度、执行器状态、系统模式；4 个按键完成模式切换、手动控制、阈值设置
- **云端连接**：通过 ESP8266 WiFi 模块连接 OneNET 物平台，每 30 秒上报数据，可接收手机 APP 下发的远程控制命令
- **数据记录**：AT24C02 EEPROM 循环存储历史数据 + 掉电保存阈值配置

### 1.2 硬件拓扑

| 外设 | 接口类型 | STM32 引脚 | 备注 |
|------|----------|-----------|------|
| SSD1306 OLED (0.96") | I2C1 | PB6 SCL, PB7 SDA | 地址 0x78 |
| AT24C02 EEPROM (256B) | I2C1 (同总线) | PB6 SCL, PB7 SDA | 地址 0xA0 |
| FC-28 土壤湿度传感器 | ADC1_IN0 | PA0 | 电压反比于湿度 |
| DS18B20 温度传感器 | One-Wire (GPIO) | PA4 | 开漏 + 外部上拉 |
| ESP8266 WiFi 模块 | USART1 | PA9 TX, PA10 RX | 115200 baud |
| 2 路继电器 (水泵/风扇) | GPIO 输出 | PA1, PA2 | 高电平有效 |
| 4 按键 (K1-K4) | GPIO 输入 | PB12-PB15 | 上拉，按下为低 |
| 蜂鸣器 | GPIO 输出 | PA3 | 高电平触发 |
| 4 LED (板载+RGB) | GPIO 输出 | PC13, PA7, PB0, PB1 | 板载低有效，其余高有效 |
| ESP8266 复位 | GPIO 输出 | PB10 | 低电平复位 |

一个关键的硬件设计决策：**OLED 和 AT24C02 共享同一路 I2C1 总线**，通过不同的从机地址区分（0x78 vs 0xA0），省下一组 IO。

### 1.3 软件架构

代码严格分两层，体现在目录结构上：

```
Core/           ← CubeMX 自动生成（HAL 初始化代码，仅 main.c USER CODE 块可手改）
User/Inc/       ← 业务模块头文件 (15 个 .h)
User/Src/       ← 业务模块源文件 (15 个 .c)
```

按职责分为三层：

```
┌─────────────────────────────────────────┐
│  应用层                                  │
│  control.c  │  key.c                    │
│  (决策大脑)  │  (按键事件)                │
├─────────────────────────────────────────┤
│  通信层                                  │
│  oled.c  │  esp8266.c  │  data_store.c  │
│  (显示)   │  (WiFi/MQTT) │  (持久化)      │
├─────────────────────────────────────────┤
│  驱动层                                  │
│  soil.c  │  ds18b20.c  │  at24c02.c     │
│  (ADC)    │  (One-Wire)  │  (I2C EEPROM) │
│  led.c   │  relay.c    │  beep.c        │
│  (GPIO)   │  (GPIO)      │  (GPIO)       │
└─────────────────────────────────────────┘
```

### 1.4 主循环数据流

`main.c` 中超级循环的调用顺序不是随意的，而是按照**数据依赖关系**精心排列：

```
key_scan()              ← 第1步：捕获用户按键（10ms 消抖）
    ↓
control_key_handler()   ← 第2步：立即消费按键事件
    ↓
ds18b20_task()          ← 第3步：触发温度转换/读取（非阻塞状态机）
    ↓
control_task()          ← 第4步：读取最新传感器 → 滞回决策 → 驱动执行器
    ↓
esp8266_task()          ← 第5步：驱动WiFi/MQTT状态机 + 上报最新 g_ctrl 数据
    ↓
oled_display_task()     ← 第6步：根据 g_ctrl 刷新显示
    ↓
data_store_task()       ← 第7步：按间隔记录历史数据到 EEPROM
```

核心思想：**每一轮循环中，按键优先捕获，传感器数据次之，控制决策基于最新数据，显示和上报消费最终状态，存储最后兜底**。

## 第二节：启动流程与 main.c 结构

### 2.1 复位启动

`startup_stm32f103xb.s` 定义了 Cortex-M3 的标准启动序列：

```
1. 硬件上电 → PC = 0x08000000 (Flash 基址)
2. 向量表[0] = _estack = 0x20005000 (20KB RAM 栈顶) → 加载到 SP
3. 向量表[1] = Reset_Handler 地址 → 加载到 PC
4. Reset_Handler:
   a. 调用 SystemInit() — 设置 Flash 等待周期、配置 HSE/HIS
   b. 拷贝 .data 段 (Flash → SRAM)：初始化已赋值的全局变量
   c. 清零 .bss 段：零初始化未赋值的全局变量
   d. 调用 main()
```

### 2.2 main() 初始化顺序

```c
// === HAL 层（CubeMX 生成）===
HAL_Init();                 // SysTick 1ms + NVIC 分组
SystemClock_Config();       // HSE 8MHz × 9 = 72MHz, APB1=36MHz, APB2=72MHz
MX_GPIO_Init();             // 所有引脚配置
MX_ADC1_Init();             // ADC1 12MHz, 单通道 PA0
MX_USART1_UART_Init();      // 115200-8-N-1
MX_I2C1_Init();             // 100kHz 标准模式

// === 用户模块（注意顺序有依赖）===
led_init();                 // 先灭所有灯
key_init();                 // 记录按键初始电平
relay_init();               // 关闭所有继电器
debug_uart_init();          // 空操作（USART1 已让给 ESP8266）
soil_init();                // 记录 tick 时间戳
ds18b20_init();             // DWT 初始化 + One-Wire 复位检测
oled_init();                // SSD1306 初始化序列 + 清屏
data_store_init();          // 读 EEPROM 存储头，校验 magic
control_init();             // 从 EEPROM 加载阈值，设默认模式
esp8266_init();             // 硬件复位 ESP8266，启动 UART RX 中断
```

顺序有讲究：`oled_init` 前先 `led_init`（防止乱闪），`data_store_init` 前先 `oled_init`（如有错误可显示），`control_init` 依赖 `data_store_init` 读出的阈值，`esp8266_init` 最后启动（避免过早占用 USART1 中断）。

### 2.3 超级循环

```c
while (1) {
    key_scan();              // 10ms period
    control_key_handler();   // 消费键事件
    ds18b20_task();          // 2s period, non-blocking SM
    control_task();          // 500ms period
    esp8266_task();          // event-driven SM
    oled_display_task();     // 500ms period
    data_store_task();       // 5min period
}
```

每个 task 内部都通过 `HAL_GetTick()` 做时间门控，**没有 `HAL_Delay()` 阻塞调用**。这是项目的硬约束。

### 2.4 中断回调

`main.c` 的 `USER CODE BEGIN 4` 区实现了两个关键的中转：

```c
// USART1 RX 中断 → ESP8266 接收
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    if (huart->Instance == USART1) {
        esp8266_rx_callback();
    }
}

// USART1 错误中断 → 同样转发（帧错误等恢复）
void HAL_UART_ErrorCallback(UART_HandleTypeDef *huart) {
    if (huart->Instance == USART1) {
        esp8266_rx_callback();
    }
}
```

`HAL_GPIO_EXTI_Callback` 当前为空 —— 按键检测完全走轮询，不依赖 EXTI。

## 第三节：驱动层 — 5 个外设模块

### 3.1 LED 模块 (`led.c`)

**设计模式：硬件映射表**

```c
typedef struct {
    GPIO_TypeDef *port;
    uint16_t      pin;
    GPIO_PinState on_state;   // 高有效 vs 低有效
    GPIO_PinState off_state;
} led_hw_t;

static const led_hw_t g_led_map[LED_COUNT] = {
    [LED_BOARD] = {LED0_GPIO_Port,   LED0_Pin,   GPIO_PIN_RESET, GPIO_PIN_SET},
    [LED_BLUE]  = {LED_BLUE_GPIO_Port, LED_BLUE_Pin, GPIO_PIN_SET, GPIO_PIN_RESET},
    [LED_RED]   = {LED_RED_GPIO_Port,  LED_RED_Pin,  GPIO_PIN_SET, GPIO_PIN_RESET},
    [LED_GREEN] = {LED_GREEN_GPIO_Port, LED_GREEN_Pin, GPIO_PIN_SET, GPIO_PIN_RESET},
};
```

通过表驱动方式，每个逻辑 LED 只需知道自己的 `on_state`/`off_state`，无需在业务代码中区分极性。所有公开 API (`led_on`/`led_off`/`led_toggle`) 都通过私有函数 `led_write()` 统一处理越界检查和 GPIO 写入。

板载 LED (PC13) 是低电平有效 (`on_state = GPIO_PIN_RESET`)，外部 RGB 三颗 (PA7/PB0/PB1) 是高电平有效 (`on_state = GPIO_PIN_SET`)。映射表优雅地统一了这两种极性差异。

### 3.2 蜂鸣器模块 (`beep.c`)

最简模块，约 65 行实现。直接使用 CubeMX 导出的 `BEEP_GPIO_Port`/`BEEP_Pin` 宏操作 PA3：

```c
void beep_on(void)    { HAL_GPIO_WritePin(BEEP_GPIO_Port, BEEP_Pin, GPIO_PIN_SET); }
void beep_off(void)   { HAL_GPIO_WritePin(BEEP_GPIO_Port, BEEP_Pin, GPIO_PIN_RESET); }
void beep_toggle(void){ HAL_GPIO_TogglePin(BEEP_GPIO_Port, BEEP_Pin); }
```

没有采用硬件映射表模式（与 LED/Relay 风格不一致），因为只有单一设备，加抽象层反而过度设计。这种"够用即止"的判断是好的工程实践。

### 3.3 继电器模块 (`relay.c`)

与 LED 相同的表驱动模式，驱动 PA1（水泵）、PA2（风扇），均为高电平触发：

```c
static const relay_hw_t g_relay_map[RELAY_COUNT] = {
    [RELAY_PUMP] = {RELAY_PUMP_GPIO_Port, RELAY_PUMP_Pin, GPIO_PIN_SET, GPIO_PIN_RESET},
    [RELAY_FAN]  = {RELAY_FAN_GPIO_Port,  RELAY_FAN_Pin,  GPIO_PIN_SET, GPIO_PIN_RESET},
};
```

所有函数入口做越界检查：

```c
void relay_on(relay_id_t relay) {
    if (relay >= RELAY_COUNT) return;
    HAL_GPIO_WritePin(g_relay_map[relay].port, g_relay_map[relay].pin,
                      g_relay_map[relay].on_state);
}
```

**历史注意点**：V1.2.0 中因 PCB 电路缺陷，所有 `relay_on/off` 调用被注释，系统仅显示逻辑状态而不实际驱动硬件。V1.3.5 硬件修复后恢复了全部 9 处 `relay_on/off` 调用。

### 3.4 土壤湿度模块 (`soil.c`)

**ADC 采集**：采用最简单的 Start-Poll-Get-Stop 阻塞模式：

```c
uint16_t soil_read_raw(void) {
    HAL_ADC_Start(&hadc1);
    HAL_ADC_PollForConversion(&hadc1, 10);  // 10ms 超时
    uint16_t val = HAL_ADC_GetValue(&hadc1);
    HAL_ADC_Stop(&hadc1);
    return val;
}
```

每次采样走完完整周期，`soil_read_avg(n)` 调用 N 次取平均。无 DMA，无连续模式。单次 10ms 阻塞在主循环中可以接受（总循环时间约 10-20ms），但 N 次采样会线性倍增加阻塞时间。

**ADC→百分比映射**：

```c
uint8_t soil_adc_to_percent(uint16_t adc_val) {
    if (adc_val >= 4095) return 0;
    // 反相线性映射：0 = 100% (wet), 4095 = 0% (dry)
    uint16_t pct = (4095 - adc_val) * 100 / 4095;
    return (pct > 100) ? 100 : (uint8_t)pct;
}
```

FC-28 传感器的输出电压与土壤含水量成反比（电阻分压原理：土壤越湿 → 导电性越好 → 传感器输出电压越低 → ADC 读数越小），因此需要反相映射。

### 3.5 DS18B20 温度传感器 (`ds18b20.c`)

全项目**驱动层最复杂的模块**，分三个子层实现。

#### Part A：DWT 微秒延时系统

```c
static void dwt_init(void) {
    CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;  // 使能 DWT 单元
    DWT->CYCCNT = 0U;
    DWT->CTRL |= DWT_CTRL_CYCCNTENA_Msk;             // 启动周期计数器
}

static void delay_us(uint32_t us) {
    uint32_t start = DWT->CYCCNT;
    uint32_t ticks = us * (SystemCoreClock / 1000000U);  // 72MHz → 每微秒72周期
    while ((DWT->CYCCNT - start) < ticks) {}
}
```

为什么不用 SysTick？SysTick 已被 HAL 库用于 1ms 时基，不具备微秒级精度。DWT（Data Watchpoint and Trace）是 Cortex-M3 调试单元中的硬件周期计数器，精度达 1/72μs（约 14ns）。

#### Part B：One-Wire 协议位操作

GPIO 配置为**开漏输出** (`GPIO_MODE_OUTPUT_OD`)：

```c
// GPIO 初始化（在 ds18b20_init 中）
GPIO_InitStruct.Pin   = DS18B20_GPIO_PIN;
GPIO_InitStruct.Mode  = GPIO_MODE_OUTPUT_OD;  // 开漏！
GPIO_InitStruct.Pull  = GPIO_NOPULL;
GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_HIGH;
```

开漏 + 外部上拉电阻是 One-Wire 的标准接法：
- "拉低" = 输出 0，NMOS 导通，总线接地
- "释放" = 输出 1，NMOS 截止，总线被外部上拉电阻拉到高电平
- "读取" = 释放总线后读 GPIO 输入电平

时序参数（标准模式，来自 DS18B20 datasheet）：

| 操作 | 时序分解 |
|------|---------|
| 复位脉冲 | 拉低 480μs → 释放 70μs → 检测应答（从机拉低）→ 等待 410μs |
| 写 1 | 拉低 6μs + 释放 64μs |
| 写 0 | 拉低 60μs + 释放 10μs |
| 读位 | 拉低 6μs → 释放 9μs → 采样 → 等待 55μs |

每个字节 LSB 先出（Least Significant Bit first）。

**关键设计决策**：所有位操作期间关闭全局中断 (`__disable_irq()` / `__enable_irq()`)。因为 One-Wire 对时序极其敏感，中断打断会导致采样窗口偏移 → CRC 错误 → 读取失败。每次读取最多关闭中断约 2ms，对系统影响可接受。

#### Part C：非阻塞状态机

```c
enum {
    DS18B20_STATE_IDLE,        // 空闲，每2秒触发转换
    DS18B20_STATE_CONVERTING   // 等待转换完成（750ms @ 12-bit）
};

// 缓存机制
static int16_t g_cached_raw;   // 最后一次成功读取的 Q8.4 原始值
static uint8_t g_data_valid;   // 至少成功读取过一次
```

工作流程：
1. **IDLE 态**：每 2 秒发 Skip ROM (0xCC) + Convert T (0x44)，转到 CONVERTING
2. **CONVERTING** 态：等待 750ms 后发 Skip ROM + Read Scratchpad (0xBE)，读 9 字节，CRC8 校验，缓存成功值

CRC8 采用 Dallas/Maxim 多项式 0x8C (x⁸+x⁵+x⁴+1)，与 DS18B20 datasheet 一致：

```c
static uint8_t crc8_byte(uint8_t crc, uint8_t data) {
    for (uint8_t i = 0; i < 8; i++) {
        uint8_t mix = (crc ^ data) & 0x01;
        crc >>= 1;
        if (mix) crc ^= 0x8C;
        data >>= 1;
    }
    return crc;
}
```

温度换算：`float temperature = raw / 16.0f`，其中 raw 为 Q8.4 定点数（高 8 位整数，低 4 位小数），默认 12 位分辨率 → 0.0625°C/LSB。

**对外接口约定**：
- `ds18b20_task()` — 非阻塞，供主循环周期性调用
- `ds18b20_get_cached_raw()` — 返回缓存值（不触发新转换）
- `ds18b20_is_valid()` — 返回是否有有效数据
- `control.c` 中只读缓存，不直接调用 `ds18b20_read_temp()`

## 第四节：通信层 — 5 个模块

### 4.1 AT24C02 EEPROM 驱动 (`at24c02.c`)

**关键参数**：I2C 地址 0xA0（7 位地址 0x50 左移 1 位），页大小 8 字节，总容量 256 字节。

**页边界感知多字节写入**（这个模块的精华）：

```c
HAL_StatusTypeDef at24c02_write_bytes(uint8_t mem_addr, const uint8_t *data, uint8_t len) {
    uint8_t offset = 0;
    uint8_t remaining = len;

    while (remaining > 0) {
        // 计算当前地址在页内的偏移位置
        uint8_t page_offset = (mem_addr + offset) % AT24C02_PAGE_SIZE; // % 8
        uint8_t chunk = AT24C02_PAGE_SIZE - page_offset;  // 本页还能写多少字节
        if (chunk > remaining) chunk = remaining;

        HAL_I2C_Mem_Write(&hi2c1, AT24C02_ADDR, mem_addr + offset,
                          I2C_MEMADD_SIZE_8BIT, (uint8_t *)data + offset, chunk, 100);

        HAL_Delay(5);  // 等待 EEPROM 内部写入周期完成（5ms max）

        offset += chunk;
        remaining -= chunk;
    }
    return HAL_OK;
}
```

为什么必须拆页？AT24C02 内部按 8 字节组织页，跨页写入时地址计数器会**回绕到页首**而不是递增。例如，向地址 0x06 写入 4 字节，后 2 字节会覆盖 0x00-0x01 而非写入 0x08-0x09。因此必须拆分跨页写入为多次 I2C 传输，每次传输不跨页。

单字节写入后延时 5ms 等待内部写入周期完成。AT24C02 标准写入周期最大 5ms。

### 4.2 数据存储模块 (`data_store.c`)

**EEPROM 完整布局**（256 字节的 AT24C02）：

```
0x00 ┌──────────────────┐
     │  magic (0xA5)    │  ← 存储头，校验数据区有效性
0x01 │  write_index     │  ← 下一条记录写入位置 (0-58)
0x02 │  record_count    │  ← 当前有效记录数 (0-59)
0x03 │  reserved        │
0x04 ├──────────────────┤
     │  record[0]       │  ← 每条记录 4 字节：
     │  record[1]       │     temp_raw_hi, temp_raw_lo,
     │  ...             │     soil_adc_hi, soil_adc_lo
     │  record[58]      │
0xEC ├──────────────────┤
     │  (8 bytes free)  │
0xF4 ├──────────────────┤
     │  magic (0xD1)    │  ← 阈值区 magic
0xF5 │  plant_type      │  ← 0=耐旱, 1=常规, 2=喜湿
0xF6 │  soil_low        │  ← 湿度下限 (%)
0xF7 │  soil_high       │  ← 湿度上限 (%)
0xF8 │  temp_low        │  ← 温度下限 (°C)
0xF9 │  temp_high       │  ← 温度上限 (°C)
0xFA │  (6 bytes free)  │
0xFF └──────────────────┘
```

**循环缓冲逻辑**：
- 最多 59 条记录（236 字节在 0x04-0xEB）
- 写满后 write_index 回到 0，覆盖最旧数据
- `data_store_dump()` 通过计算线性起始位置还原时间顺序

**阈值存储**：
- 放在 0xF4-0xF9（EEPROM 末端），与数据区隔离
- 独立 magic (0xD1)，与数据区 magic (0xA5) 分开校验
- `control_init()` → `data_store_load_thresholds()` → magic 不匹配返回默认值
- 设置模式 K4 保存 → `data_store_save_thresholds()`

**阈值配置结构**（V1.3.5 从 6 字段简化为 5 字段）：

```c
typedef struct {
    uint8_t plant_type;  // 0=耐旱, 1=常规, 2=喜湿
    uint8_t soil_low;    // 湿度下限（触发灌溉）
    uint8_t soil_high;   // 湿度上限（停止灌溉，滞回）
    uint8_t temp_low;    // 温度下限（触发状态变化）
    uint8_t temp_high;   // 温度上限（触发散热）
} threshold_config_t;
```

### 4.4 ESP8266 WiFi 通信模块 (`esp8266.c`)

全项目最复杂模块（~700 行），核心是**三层嵌套状态机**。

#### Layer 1：AT 命令引擎

```c
// 发送命令 + 记录期望响应
static void at_send(const char *cmd, const char *expect, uint32_t timeout_ms) {
    rx_buf_clear();
    HAL_UART_Transmit(&huart1, (uint8_t *)cmd, strlen(cmd), 100);
    g_at_expect = expect;
    g_at_timeout = timeout_ms;
    g_at_start = HAL_GetTick();
    g_at_state = AT_WAIT_RESP;
}

// 检查响应（返回值：0=等待中, 1=匹配成功, 2=超时）
static int at_check(void) {
    if (HAL_GetTick() - g_at_start > g_at_timeout) return 2;  // timeout
    if (strstr(g_rx_buf, g_at_expect)) return 1;               // matched
    return 0;                                                   // waiting
}
```

典型的请求-响应模式。`at_check` 每次被 `esp8266_task` 调用时检查一次，不阻塞。局限性：只能检测正匹配或超时，不能识别 `ERROR` 等负面响应。

#### Layer 2：19 态连接状态机

```
            ESP_CONN_IDLE
                 ↓
            ESP_CONN_HW_RESET  ← 拉低 RST 100ms
                 ↓
           ESP_CONN_WAIT_READY ← 等待 2000ms 启动
                 ↓
           ESP_CONN_AT_TEST    ← 发送 AT，最多重试 3 次 → 失败 → ESP_CONN_ERROR
                 ↓
            ESP_CONN_ATE0      ← 关闭回显
                 ↓
           ESP_CONN_CWMODE     ← 设置 Station 模式 (CWMODE=1)
                 ↓
            ESP_CONN_CWJAP     ← 连接 WiFi AP，15s 超时，最多重试 2 次
                 ↓
           ESP_CONN_WIFI_OK    ← WiFi 连接成功
                 ↓
        ESP_CONN_MQTT_USERCFG  ← MQTT 用户证书配置，6s 超时
                 ↓
         ESP_CONN_MQTT_CONN    ← 连接 MQTT Broker，15s 超时，最多重试 2 次
                 ↓
         ESP_CONN_MQTT_WAIT    ← 稳定延时 500ms
                 ↓
          ESP_CONN_MQTT_SUB1   ← 订阅 property/post/reply 主题 (重试2次，失败继续)
                 ↓
          ESP_CONN_MQTT_SUB2   ← 订阅 property/set 主题 (重试2次，失败继续)
                 ↓
          ESP_CONN_MQTT_OK     ← ═══ 运行态 ═══
               ↓    ↑                         ↓
        ESP_PUB_CMD → ESP_PUB_DATA   ESP_REPLY_CMD → ESP_REPLY_DATA
               ↓    ↑                         ↓
               └────┘                         └──→ 返回 MQTT_OK

      任何状态检测到断线 → ESP_CONN_ERROR → 等待 5s → ESP_CONN_HW_RESET
```

每个状态在 `esp8266_task` 的一次调用中执行一步。发送 AT 命令后转换到等待态，下一次调用时检测响应。失败后大部分状态导向 `ESP_CONN_ERROR`，5 秒后硬件复位从头开始。

#### Layer 3：MQTT 数据面

**MQTT 数据上报**（`mqtt_build_payload`）— 每 30 秒触发一次：

```json
{
  "id": "123",
  "params": {
    "temperature":  {"value": 25.3},
    "soil_moisture": {"value": 65},
    "pump_state":    {"value": true},
    "fan_state":     {"value": false},
    "mode":          {"value": 0},
    "status":        {"value": 0}
  }
}
```

温度值通过整数拆分避免使用 `%f`（STM32 的 `nano.specs` 默认不支持 `%f` 格式化）：

```c
int16_t temp_raw = g_ctrl.temp_raw;
int16_t temp_int = temp_raw / 16;         // 整数部分
int16_t temp_frac = ((temp_raw % 16) * 10) / 16;  // 小数第一位
```

`pump_state` 和 `fan_state` 使用 JSON 的 `true`/`false`（而非 `1`/`0`），这是 OneNET 物模型平台的类型要求。

**MQTT 下行命令解析**（`mqtt_check_downlink`）：

```c
static void mqtt_check_downlink(void) {
    char *sub = strstr(g_rx_buf, "+MQTTSUBRECV:");
    if (!sub) return;
    char *set = strstr(sub, "property/set");
    if (!set) return;

    // 解析 mode 字段
    char *mode = strstr(set, "\"mode\":");
    if (mode) { g_ctrl.mode = mqtt_parse_value(mode); }

    // 解析 pump_state 字段
    char *pump = strstr(set, "\"pump_state\":");
    if (pump) {
        g_ctrl.pump_on = mqtt_parse_bool(pump);
        if (g_ctrl.mode == SYS_MODE_AUTO) g_ctrl.mode = SYS_MODE_MANUAL;
    }

    // 解析 fan_state 字段
    char *fan = strstr(set, "\"fan_state\":");
    if (fan) {
        g_ctrl.fan_on = mqtt_parse_bool(fan);
        if (g_ctrl.mode == SYS_MODE_AUTO) g_ctrl.mode = SYS_MODE_MANUAL;
    }
}
```

使用 `strstr` 在 RX 缓冲区中搜索关键字 + `atoi` 解析数值，无完整 JSON 解析器。

**远程控制的安全设计**：远程修改 `pump_state` 或 `fan_state` 时会自动将模式从 AUTO 切换到 MANUAL，防止自动控制逻辑立即覆盖用户的远程指令。

**断线检测**（`mqtt_check_disconnect`）：在所有活跃状态下扫描 RX 缓冲区中的 `+MQTTDISCONNECTED:` 或 `WIFI DISCONNECT`，发现即触发重连：

```c
static int mqtt_check_disconnect(void) {
    if (strstr(g_rx_buf, "+MQTTDISCONNECTED:") ||
        strstr(g_rx_buf, "WIFI DISCONNECT")) {
        rx_buf_clear();
        g_conn_state = ESP_CONN_ERROR;
        return 1;
    }
    return 0;
}
```

**RX 线性缓冲区**：

```c
#define ESP_RX_BUF_SIZE 512
static char g_rx_buf[ESP_RX_BUF_SIZE];
static volatile uint16_t g_rx_len;  // ISR 写入，主循环读取
```

- UART RX 中断逐字节接收，`esp8266_rx_callback()` 追加到 `g_rx_buf`
- `rx_buf_clear()` 使用 `__disable_irq()` / `__enable_irq()` 原子清零
- 缓冲满 512 字节后丢弃后续数据。AT 响应通常 < 512 字节，此限制可接受

## 第五节：应用层

### 5.1 按键模块 (`key.c`)

4 个按键（PB12-PB15），均为上拉输入，按下读到低电平 (`GPIO_PIN_RESET`)。

**消抖状态机**（每个按键独立的 7 字段状态结构）：

```c
typedef struct {
    uint8_t last_raw_pressed;      // 最近一次读到的原始电平值
    uint8_t stable_pressed;        // 消抖后的稳定状态
    uint8_t pressed_event_pending; // 有待消费的短按事件
    uint8_t long_event_pending;    // 有待消费的长按事件 (仅 KEY_1)
    uint8_t long_fired;            // 长按已触发 (抑制释放时的短按)
    uint32_t last_change_tick;     // 最后一次电平变化的时间戳
    uint32_t press_start_tick;     // 按下确认的起始时间戳
} key_state_t;
```

**消抖逻辑**（`key_scan()`，每 10ms 调用）：

1. 读 GPIO 电平，与 `last_raw_pressed` 比较
2. 如果不同：更新 `last_raw_pressed` 和 `last_change_tick`（电平变化检测）
3. 如果连续 30ms（3 次扫描）电平不变：更新 `stable_pressed` 为稳定值
4. 稳定按下时：设 `pressed_event_pending`，记录 `press_start_tick`
5. 对 KEY_1：如果稳定按下持续 2000ms，设 `long_event_pending` + `long_fired`
6. KEY_1 释放时：如果 `long_fired` 已置位，**不产生短按事件**（抑制误触发）

**事件消费**：`key_get_event()` 按优先级返回事件（KEY_1 长按优先），消费后清除对应标志。

```c
key_event_t key_get_event(void) {
    // 优先检查长按事件
    for (int i = 0; i < KEY_ID_COUNT; i++) {
        if (g_key_state[i].long_event_pending) {
            g_key_state[i].long_event_pending = 0;
            return KEY_EVENT_1_LONG + i;
        }
    }
    // 再检查短按事件
    for (int i = 0; i < KEY_ID_COUNT; i++) {
        if (g_key_state[i].pressed_event_pending) {
            g_key_state[i].pressed_event_pending = 0;
            return KEY_EVENT_1_PRESSED + i;
        }
    }
    return KEY_EVENT_NONE;
}
```

### 5.2 控制模块 (`control.c`)

系统的"大脑"，约 400 行实现。负责全部的运行模式管理、传感器阈值决策、执行器驱动、报警和 LED/蜂鸣器指示。

#### 全局状态 `g_ctrl`

```c
volatile control_state_t g_ctrl = {
    .mode         = SYS_MODE_AUTO,
    .status       = SYS_STATUS_NORMAL,
    .pump_on      = 0,
    .fan_on       = 0,
    .alarm_active = 0,
    .alarm_muted  = 0,
    .temp_raw     = 0,
    .soil_val     = 0,
    .soil_pct     = 0,
    .thresh       = {1, 30, 60, 15, 35},  // 默认：常规植物
    .setting_index = 0,
};
```

声明为 `volatile` 是因为 ESP8266 的下行解析会直接写入 `g_ctrl.pump_on`/`g_ctrl.fan_on`/`g_ctrl.mode`，编译器不能优化掉对"看起来没变化"的变量的读取。

#### 自动控制逻辑 (`control_auto_logic`，每 500ms 执行)

```c
static void control_auto_logic(void) {
    // 1. 土壤/水泵回路（滞回区间：soil_low ~ soil_high）
    if (g_ctrl.soil_pct < g_ctrl.thresh.soil_low) {
        relay_on(RELAY_PUMP);        // 太干 → 开启灌溉
    } else if (g_ctrl.soil_pct >= g_ctrl.thresh.soil_high) {
        relay_off(RELAY_PUMP);       // 够湿 → 停止灌溉
    }

    // 2. 温度/风扇回路（滞回：temp_high - 2°C ~ temp_high）
    int16_t temp_high_raw = g_ctrl.thresh.temp_high * 16;
    if (g_ctrl.temp_raw > temp_high_raw) {
        relay_on(RELAY_FAN);         // 太热 → 开启散热
    } else if (g_ctrl.temp_raw <= temp_high_raw - 32) {  // -32 = -2.0°C
        relay_off(RELAY_FAN);        // 凉快 → 停止散热
    }

    // 3. 状态判定（DRY 优先级高于 HOT）
    if (g_ctrl.soil_pct < g_ctrl.thresh.soil_high) {
        g_ctrl.status = SYS_STATUS_DRY;
    } else if (g_ctrl.temp_raw > g_ctrl.thresh.temp_low * 16) {
        g_ctrl.status = SYS_STATUS_HOT;
    } else {
        g_ctrl.status = SYS_STATUS_NORMAL;
    }

    // 4. 报警判定（需要执行动作时触发）
    g_ctrl.alarm_active = (g_ctrl.soil_pct < g_ctrl.thresh.soil_low)
                       || (g_ctrl.temp_raw > temp_high_raw);
}
```

**滞回设计详解**：

- **水泵**：开启条件 `soil < soil_low`（如 30%），关闭条件 `soil >= soil_high`（如 60%）。滞回区间 30%-60% 防止频繁切换
- **风扇**：开启条件 `temp > temp_high`（如 35°C），关闭条件 `temp <= temp_high - 2°C`（如 33°C）。2°C 滞回防止温度轻微波动引起的风扇抖动

两个控制回路**完全独立**，可同时工作（水泵 + 风扇可能同时开启）。

**状态优先级**：DRY > HOT > NORMAL。即土壤缺水时优先显示缺水状态，土壤充足后才检查温度。

#### 植物预设

```c
static const threshold_config_t PRESETS[3] = {
    {0, 20, 40, 10, 38},  // 耐旱植物：湿度 20-40%, 温度 10-38°C
    {1, 30, 60, 15, 35},  // 常规植物：湿度 30-60%, 温度 15-35°C
```

#### 报警与 LED 指示

```c
static void control_update_led(void) {
    // 设置模式：蓝灯常亮
    if (g_ctrl.mode == SYS_MODE_SETTINGS) {
        led_on(LED_BLUE); led_off(LED_RED); led_off(LED_GREEN); return;
    }

    switch (g_ctrl.status) {
    case SYS_STATUS_NORMAL:
        led_off(LED_RED); led_off(LED_BLUE);
        led_on(LED_GREEN);   // 正常 = 绿灯常亮
        break;
    case SYS_STATUS_DRY:
        led_off(LED_GREEN); led_off(LED_BLUE);
        if (g_ctrl.alarm_active) led_toggle(LED_RED);  // 报警 = 红灯闪烁
        else                     led_on(LED_RED);       // 警告 = 红灯常亮
        break;
    case SYS_STATUS_HOT:
        led_off(LED_GREEN); led_off(LED_RED);
        if (g_ctrl.alarm_active) led_toggle(LED_BLUE);  // 报警 = 蓝灯闪烁
        else                     led_on(LED_BLUE);       // 警告 = 蓝灯常亮
        break;
    }
}
```

LED 分级：**警告**（常亮）vs **报警**（闪烁）。缺水对应红灯，高温对应蓝灯，正常对应绿灯。

**蜂鸣器报警**：300ms 周期翻转 (`beep_toggle`)，实现间歇鸣叫。K4 静音带 60 秒超时自动恢复：

```c
// 静音超时检查
if (g_ctrl.alarm_muted && HAL_GetTick() - alarm_mute_start >= 60000) {
    g_ctrl.alarm_muted = 0;  // 60秒后自动恢复报警
}
```

状态变化时自动恢复报警（`alarm_muted = 0`），防止静音后状态恶化而用户不知情。

#### 按键处理 (`control_key_handler`)

设置模式：
- K1 短按：游标 `setting_index` 循环 0-4（植物类型 → 湿度下限 → 湿度上限 → 温度下限 → 温度上限）
- K2 短按：减少当前项的值（带边界保护，步进 ±1% / ±1°C）
- K3 短按：增加当前项的值
- K4 短按：`setting_buf` → `g_ctrl.thresh` → `data_store_save_thresholds()` → 返回 AUTO

运行模式 AUTO：
- K1 短按：切换到 MANUAL
- K1 长按 2s：拷贝当前阈值到 `setting_buf` → 进入 SETTINGS
- K4 短按：静音 / 取消静音

运行模式 MANUAL：
- K1 短按：切换回 AUTO
- K2 短按：水泵 ON/OFF 切换
- K3 短按：风扇 ON/OFF 切换
- K4 短按：静音 / 取消静音

### 6.3 并发安全性

超级循环是**单线程**的，不存在抢占式并发。唯一的"类并发"来源于中断：

- ESP8266 UART RX 中断 (`esp8266_rx_callback`) → 写入 `g_rx_buf[512]` + 递增 `g_rx_len`
- 主循环 `esp8266_task()` → 读取 `g_rx_buf` 进行 AT 响应匹配和 MQTT 解析

关键保护：`rx_buf_clear()` 使用 `__disable_irq()` / `__enable_irq()` 原子清空缓冲区，防止 ISR 和主循环同时修改 `g_rx_len`。

```c
static void rx_buf_clear(void) {
    __disable_irq();
    g_rx_len = 0;
    memset(g_rx_buf, 0, ESP_RX_BUF_SIZE);
    __enable_irq();
}
```

ESP8266 下行解析直接写 `g_ctrl` 的 pump/fan/mode 字段后，主循环中 `control_task` 的下一次调用就会消费这些更新。由于单线程模型，不存在读写冲突。

## 第七节：关键设计决策与权衡

### 7.1 超级循环 vs RTOS

**选择超级循环，原因**：

- STM32F103C8T6 仅 20KB RAM / 64KB Flash。RTOS 内核本身占 ~5-8KB Flash + 每任务 ~200B 栈空间
- 任务只有 7 个，全部是短周期非阻塞任务，RTOS 的抢占式调度价值有限
- 超级循环的行为完全可预测：固定的执行顺序、无优先级反转、无死锁风险
- 课程设计/教学场景下，简单性 > 工程完备性

### 7.2 单 I2C 总线共享 OLED + EEPROM

两个设备挂在同一路 I2C1（PB6/PB7），地址不冲突（OLED 0x78 vs AT24C02 0xA0）。省下一组 IO（不需要第二路 I2C），简化 PCB 布线。代价是两者的 I2C 事务串行化（同一时刻只能与一个设备通信），但 OLED 刷新 500ms、EEPROM 写入 5min 的间隔下完全不影响系统体验。

### 7.3 DS18B20 关中断

One-Wire 是**位操作级时序协议**，微秒级的窗口必须精确命中。HAL 库的 SysTick 中断（1ms 周期）和其他中断可能在位操作期间打断，导致采样窗口偏移 → CRC 错误 → 读取失败。

关闭全局中断约 ~2ms/次（复位 960μs + 9 字节读取），在最坏情况下增加中断延迟 2ms，对 10ms 按键扫描和 UART 中断缓冲区（512 字节，115200 baud 下填满约 44ms）都不构成实质性影响。

### 7.4 字符串 JSON 解析 vs 完整 JSON 库

用 `strstr` + `atoi` 解析 OneNET 下行命令，而非引入 cJSON 等完整 JSON 库。原因：

- cJSON 占约 5-10KB Flash，在 64KB 总量中占比显著（8-16%）
- OneNET 下发格式是**固定的机器生成 JSON**，字段名和位置可预测
- 简单字符串解析无动态内存分配，不存在 malloc 失败丢帧的问题
- 嵌入式场景中，"够用"比"完备"更重要

### 7.5 `g_ctrl` 的 `volatile` 修饰

ESP8266 下行解析 `mqtt_check_downlink()` 直接写 `g_ctrl.mode` / `g_ctrl.pump_on` / `g_ctrl.fan_on`。如果不用 `volatile`，编译器可能认为主循环中 `g_ctrl` 没有被"显式"修改（因为修改在函数指针/间接调用链路中不明显），将其读取优化为寄存器缓存，导致远程控制命令被"吞掉"。

`volatile` 确保每次访问都从内存重新读取，牺牲少量性能换取正确性。

### 7.6 debug_uart 保留空壳

USART1 让给 ESP8266 后，调试输出功能丧失。但保留了所有函数签名和注释代码，方便未来：

- 启用 USART2 或 USART3 恢复调试输出
- 使用 SWO (Serial Wire Output) 做 printf 重定向
- 或在 ESP8266 未插入时切换回调试模式

这体现了"可恢复性"的设计考量——不因为当前用不到就删除代码，而是保留接口等待未来恢复。
